# 最小落地版本

本项目当前不按宏大教育制度拆分，而按可以开发、可以演示、可以逐步迭代的产品系统拆分。

第一阶段聚焦 6 个系统：

1. 形象设计系统
2. 知识系统
3. 成长系统
4. 游戏系统
5. 记忆系统
6. 硬件桥接系统

这 6 个系统构成最小可落地产品。

## 一句话 MVP

孩子画出一个属于自己的伙伴，系统把它变成统一美术风格的 2D 卡通形象，再逐步生成可交互的 3D 伙伴；孩子通过学习、互动和游戏，让伙伴与自己共同成长。

## 1. 形象设计系统

### 解决什么问题

让孩子从第一天就感到：这个伙伴是我创造出来的，不是系统发给我的标准角色。

它负责建立孩子对伙伴的第一层情感连接。

### 用户流程

1. 孩子在草稿纸上画一个卡通形象
2. 孩子描述它的性格、能力和名字
3. 系统把草稿识别为统一美术风格的 2D 卡通形象
4. 孩子确认或调整
5. 确认后生成 3D 形象
6. 3D 形象绑定骨骼
7. 预设 20 个交互动作
8. 绑定语音包

### 第一阶段做到什么

个人开发者版本先不做完整自动 3D 生成链路。

第一阶段目标：

- 支持上传孩子的草稿图
- 输入名字、性格、能力描述
- 生成统一风格的 2D 伙伴图
- 提供 3 到 5 个基础表情或姿态
- 后台先用固定模板模拟 3D 伙伴展示

### 第二阶段再做什么

- 2D 转 3D
- 骨骼绑定
- 20 个交互动作
- 语音包
- 动作和性格关联
- 伙伴房间或伙伴空间

### 最大难点

必须保证每个孩子生成的宠物美术风格一致。

如果风格不一致，产品会显得廉价、混乱，也无法形成品牌识别。

### 需要解决的技术问题

- 草稿图理解
- 统一风格重绘
- 角色一致性
- 多姿态一致性
- 2D 到 3D 的角色保持
- 骨骼绑定自动化或半自动化
- 语音风格一致性

## 2. 知识系统

### 解决什么问题

前期知识系统具有商业定性意义。

家长还没有意识到未来调用知识的成本会变得非常廉价，所以知识系统是获得家长支持的重要手段。

它让家长理解：这个产品不是单纯陪玩，也不是单纯聊天，它确实能帮助孩子学习知识。

### 产品定位

知识系统不是传统网课，也不是题库。

它要把知识变成伙伴成长、技能解锁和游戏行动的一部分。

### 第一阶段做到什么

- 孩子输入想学的主题
- 系统按年龄生成简单解释
- 系统生成一个小测验或小挑战
- 完成后伙伴获得一个知识相关的小技能或故事变化

例子：

- 学会昆虫的基本结构，伙伴获得“细节观察”能力
- 学会声音的传播，伙伴获得“回声耳朵”
- 学会简单几何，伙伴获得“形状感知”

### 对家长的价值

家长看到的是：

- 孩子确实学到了知识
- 知识解释符合年龄
- 学习不是刷题，而是和伙伴成长结合
- 孩子愿意主动学一点东西

### 注意

知识系统前期是获取家长支持的入口，但不能让产品重新变成知识灌输系统。

## 3. 成长系统

### 解决什么问题

成长系统负责记录孩子和伙伴的长期变化。

开发沟通中可以使用“人格完整指数”和“羁绊指数”这两个概念，方便团队讨论系统状态。

但现实产品里绝不可以显示这些数值。

### 内部核心指标

开发内部暂定：

- 人格完整指数
- 羁绊指数

它们只用于系统内部判断和调节，不显示给孩子、家长或外部人员。

### 产品外显形式

对孩子展示的不是分数，而是：

- 伙伴状态
- 伙伴故事
- 伙伴表情
- 伙伴动作
- 伙伴空间变化
- 新技能
- 回忆片段
- 冒险记录

### 第一阶段做到什么

- 每次互动后记录一次成长事件
- 根据事件类型更新内部成长状态
- 用一段伙伴故事变化表达成长

例子：

孩子主动学习一个知识点：

- 内部：人格完整指数小幅增加，羁绊指数小幅增加
- 外显：伙伴的背包里多了一本发光小书

孩子连续只聊天不行动：

- 内部：羁绊可能增加，但人格完整指数不增加，必要时轻微下降
- 外显：伙伴温和提醒“我们要不要把这个想法带到外面试一下”

### 红线

- 不显示数值
- 不做排名
- 不生成孩子人格评分
- 不把成长状态卖给家长或机构
- 不让家长用它考核孩子

## 4. 游戏系统

### 解决什么问题

前期为了获得较好的落地效果，游戏系统是必须的。

没有游戏性，孩子不一定愿意长期回来。

但游戏系统必须服务成长，而不是单纯制造沉迷。

### 第一阶段游戏核心

第一阶段不做复杂对战。

只做三个轻量机制：

1. 伙伴成长
2. 技能解锁
3. 小挑战

### 伙伴成长

孩子和伙伴互动、学习、完成小挑战后，伙伴发生变化。

变化可以是：

- 新表情
- 新动作
- 新道具
- 新技能
- 新故事片段
- 新房间物品

### 技能解锁

孩子学习知识或完成现实行动后，伙伴解锁技能。

技能要酷炫，但不能只靠刷。

### 小挑战

挑战可以来自知识系统或现实行动。

例如：

- 观察挑战
- 记忆挑战
- 表达挑战
- 制作挑战
- 关系挑战
- 专注挑战

### 成瘾机制的处理

必须承认产品需要吸引力。

但吸引力要导向：

- 学习
- 创造
- 互动
- 现实行动
- 回忆沉淀

不能导向：

- 无限刷内容
- 无意义点击
- 虚假数值增长
- 孩子之间攀比

## 5. 记忆系统

### 解决什么问题

记忆系统负责记录每一天的交互反馈，并在内部更新人格完整指数和羁绊指数的变化。

它的作用是保证两个核心内部指标在长期使用中能够被追踪、调节，并尽量保持正向增长。

### 记录什么

每天记录：

- 孩子说了什么主题
- 孩子学了什么知识
- 孩子是否完成小挑战
- 孩子是否表达情绪
- 孩子是否主动选择
- 孩子是否拒绝建议
- 孩子是否回到现实行动
- 伙伴与孩子发生了什么故事

### 如何更新

在原有内部状态基础上增减。

例如：

- 主动学习：人格完整指数增加，羁绊指数增加
- 主动表达情绪：羁绊指数增加
- 主动拒绝 AI 并说明原因：人格完整指数增加
- 完成现实挑战：人格完整指数增加，羁绊指数增加
- 长时间空转聊天：羁绊指数可能小幅增加，人格完整指数不增加
- 明显依赖或逃避：系统触发温和干预

### 外显方式

记忆系统不显示表格、不显示分数。

它只通过这些方式表达：

- “我还记得你上次说过喜欢声音。”
- “你的伙伴把那次发现放进了回忆柜。”
- “这次它学会了一个和你有关的新动作。”
- “你们的故事多了一页。”

## 6. 硬件桥接系统

### 解决什么问题

个人开发者早期无法自研硬件，也无法承担完整大模型成本。

硬件桥接系统的目标是：让孩子的伙伴能连接手机上现有的模型和 AI 应用能力，例如豆包、DeepSeek 等，而不是一开始就自建完整模型服务。

### 产品定位

它不是先做一个新硬件，而是先做一个“桥”。

这座桥连接：

- 孩子的伙伴形象
- 手机里的 AI 模型能力
- 本项目自己的成长、游戏和记忆系统

### 第一阶段做到什么

第一阶段先做软件桥接，不做实体硬件。

可以支持：

- 在网页中复制提示词到豆包、DeepSeek 等现有 App
- 从现有模型返回结果后，粘贴回本系统
- 本系统负责把外部模型结果转化为伙伴语言、知识挑战或故事变化
- 记录互动结果到本地记忆系统

这是最低成本版本。

### 第二阶段做到什么

如果条件允许，再做更自动化的桥接：

- 手机快捷指令
- 分享菜单接入
- 浏览器插件
- 本地小工具
- API 接入
- 语音转发

### 第三阶段再考虑硬件

只有软件桥接证明成立后，再考虑实体硬件。

可能形态：

- 桌面小伙伴屏幕
- 蓝牙语音按钮
- 儿童手持交互设备
- 带麦克风和扬声器的小终端

### 桥接原则

- 不绑定单一模型
- 优先兼容手机里已有 AI 工具
- 本项目掌握伙伴、成长、游戏和记忆逻辑
- 外部模型只负责语言、知识和推理能力
- 不能把孩子完整隐私直接暴露给外部模型
- 能本地保存的记忆先本地保存

### 为什么必须有这个系统

因为早期真正稀缺的不是模型能力，而是产品组织能力。

豆包、DeepSeek 等模型已经能回答问题，但它们不会自动变成孩子自己的长期伙伴，也不会自动形成统一形象、游戏成长、记忆故事和人格陪伴。

硬件桥接系统让本项目先借用已有模型能力，把精力集中在真正差异化的部分：

- 伙伴形象
- 成长机制
- 游戏反馈
- 记忆系统
- 儿童体验

## 第一阶段真正要做的版本

个人开发者第一版建议做成一个网页 Demo。

只做 5 个页面：

1. 创建伙伴
2. 伙伴主页
3. 知识小挑战
4. 互动聊天
5. 回忆故事本

硬件桥接第一版不单独做页面，只在聊天和知识挑战中提供“复制到外部 AI / 粘贴返回结果”的桥接入口。

## 第一版技术策略

### 不做

- 不做原生 App
- 不做真实 3D 自动生成
- 不做复杂骨骼绑定
- 不做完整语音包
- 不做支付
- 不做账号体系
- 不做家长后台
- 不做真实儿童数据采集
- 不做实体硬件
- 不做完整模型自研

### 先做

- 网页版
- 本地存储
- 固定美术风格提示词
- 2D 伙伴生成
- 模拟 3D 伙伴展示
- 简单聊天
- 知识挑战
- 隐藏成长状态
- 回忆故事本
- 外部 AI 桥接入口

## 最小闭环

```mermaid
flowchart LR
    A["孩子画伙伴"] --> B["生成统一风格 2D 形象"]
    B --> C["孩子确认伙伴"]
    C --> D["学习一个小知识"]
    D --> E["完成一个小挑战"]
    E --> F["内部更新成长和羁绊状态"]
    F --> G["伙伴获得故事变化或技能"]
    G --> H["记录到回忆故事本"]
    H --> D
```

## 当前最重要的产品判断

现在不是先验证教育制度，也不是先做数字身份。

现在先验证：

孩子亲手画出来的伙伴，能不能通过统一风格生成、知识挑战、成长反馈和记忆故事，让孩子愿意持续互动。

如果这个成立，后面再扩展游戏、3D、语音、现实任务和数字资产。
