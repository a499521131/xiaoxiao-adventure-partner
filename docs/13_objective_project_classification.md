# 客观项目分类

本文件用于把当前项目从宏大设想收缩为个人开发者可以推进的执行范围。

判断标准：

- 是否一个人可以开始做
- 是否低成本
- 是否能形成可演示结果
- 是否能证明产品差异
- 是否暂时不依赖真实儿童数据
- 是否暂时不依赖团队、资金、硬件和线下资源

## A 类：当前必须做

这些是第一版 Demo 的核心。

### A1. 形象设计系统

解决的问题：让孩子觉得伙伴是自己创造的，而不是系统分配的。

第一版范围：

- 上传孩子画的草稿
- 输入名字、性格、能力
- 生成统一美术风格的 2D 卡通伙伴
- 提供 3 到 5 个姿态或表情

客观评价：这是项目最有差异化的入口，必须优先做。

### A2. 知识系统

解决的问题：让家长理解产品不是陪聊，而是能产生学习价值。

第一版范围：

- 输入学习主题
- 生成儿童能懂的解释
- 生成一个小挑战
- 完成后触发伙伴故事变化

客观评价：商业定性需要它，但不能做成网课或题库。

### A3. 游戏系统

解决的问题：让孩子愿意回来使用。

第一版范围：

- 伙伴成长反馈
- 技能解锁
- 小挑战完成反馈

客观评价：必须有，但第一版不能做复杂对战和大地图。

### A4. 记忆系统

解决的问题：让伙伴显得持续理解孩子，而不是每次重新开始。

第一版范围：

- 本地保存孩子昵称、伙伴设定、兴趣关键词
- 保存挑战记录和故事记录
- 让伙伴在后续对话中引用过去经历

客观评价：必须做，但先用本地存储，不做账号和云同步。

### A5. 成长系统

解决的问题：让系统内部能根据互动变化调整伙伴反馈。

第一版范围：

- 内部记录人格完整指数和羁绊指数
- 不显示任何数值
- 用伙伴状态、故事变化、道具变化表达结果

客观评价：可以做轻量版。不要过度设计公式，也不要让它变成评价系统。

### A6. 硬件桥接系统

解决的问题：个人开发者不自建大模型和硬件，也能借用手机上已有 AI 能力。

第一版范围：

- 生成可复制到豆包、DeepSeek 等工具的提示词
- 支持把外部模型结果粘贴回来
- 本系统负责把结果转成伙伴语言、挑战或故事

客观评价：这个系统很现实，适合零资金阶段。第一版不要做自动化硬件。

## B 类：暂缓做

这些有价值，但现在做会拖慢落地。

### B1. 3D 形象生成与骨骼绑定

原因：技术链路长，角色一致性难，个人开发者容易卡死。

当前处理：先用 2D 和模拟 3D 展示替代。

### B2. 20 个交互动作

原因：动作、绑定、展示都需要资产制作。

当前处理：先做 3 到 5 个静态姿态或简单动画。

### B3. 语音包

原因：声音一致性、儿童安全和模型成本都需要额外处理。

当前处理：先做文字互动，后续再接 TTS。

### B4. 真实儿童试点

原因：涉及安全、隐私、家长授权和伦理边界。

当前处理：先做成人演示版和模拟流程，不采集真实儿童数据。

### B5. 家长后台

原因：容易把产品拉回监督和考核。

当前处理：第一版不做，只保留演示说明。

## C 类：归档为背景，不进入当前开发

这些是长期方向，但现在不是执行范围。

### C1. 替代学校的完整教育体系

客观评价：愿景太大，不适合当前阶段。

处理方式：保留在背景文档，不进入开发计划。

### C2. 90 天试点计划

客观评价：需要家庭、孩子和运营资源。

处理方式：等 Demo 有说服力后再重新设计。

### C3. 本地生活网络

客观评价：基础设施要求高。

处理方式：不做。

### C4. 导师网络

客观评价：组织成本高。

处理方式：不做。

### C5. 数字身份和长期资产

客观评价：战略上重要，但合规和产品成本很高。

处理方式：只保留概念，不做功能。

### C6. 商业生态和融资叙事

客观评价：现在没有产品，谈生态过早。

处理方式：不做。

## D 类：建议删除或停止维护

这里的“删除”不是立即物理删除文件，而是不再把它们当作当前项目文档维护。

### D1. 过大的项目群拆分

对应文件：[项目群拆分与协作分工](docs/11_program_breakdown.md)

原因：对个人开发者不现实，会制造压力和分散注意力。

建议：保留为历史记录，不再更新。

### D2. 过完整的教育系统架构

对应文件：

- [教育系统架构](docs/02_growth_os_architecture.md)
- [项目制学习引擎](docs/03_project_learning_engine.md)
- [评价与作品集系统](docs/04_assessment_portfolio.md)

原因：这些是未来教育体系设计，不是当前产品开发。

建议：保留为思想资料，不进入开发 backlog。

### D3. 过早的试点和线下计划

对应文件：[90 天试点计划](docs/05_pilot_plan.md)

原因：现在没有产品、资金和合规准备。

建议：暂停维护。

## 当前项目分类结果

### 当前开发范围

- 形象设计系统
- 知识系统
- 游戏系统
- 记忆系统
- 成长系统
- 硬件桥接系统

### 当前 Demo 页面

- 创建伙伴
- 伙伴主页
- 知识小挑战
- 互动聊天
- 回忆故事本

### 当前技术策略

- Web Demo
- 本地存储
- 2D 伙伴生成
- 固定风格提示词
- 外部 AI 复制粘贴桥接
- 不接真实儿童数据

### 当前验证目标

不是验证教育理念，而是验证一个产品问题：

孩子画出的伙伴，能不能通过统一风格生成、知识挑战、游戏反馈和记忆故事，形成一个有吸引力的最小体验。

## 最客观的一句话

现在这个项目不是教育系统，也不是儿童数字资产平台，而是一个“AI 伙伴生成 + 知识挑战 + 成长故事”的个人开发者 Demo。
